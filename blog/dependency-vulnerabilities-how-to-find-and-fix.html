Dependency Vulnerabilities: How to Find and Fix Them Before They Find You

March 23, 2026 • 7 min read • Security Practices

Every npm package you install in your project has its own dependencies, and those dependencies have dependencies. By the time you have a working Next.js application, you have hundreds or thousands of packages in your node_modules. Any one of them can contain a known security vulnerability. Most developers do not check.

Why Dependencies Are a Real Attack Surface

The supply chain attack vector is not theoretical. In 2024 and 2025, several high-profile incidents involved attackers compromising popular npm packages and injecting malicious code that ran in millions of applications. The packages were legitimate and widely used. The compromise happened upstream. Users who had pinned to a specific version were safe. Users who had not were not.

Even without supply chain attacks, dependencies accumulate CVEs over time. A package that was safe when you installed it may have a known vulnerability discovered six months later. If you are not monitoring your dependencies, you will not know until someone exploits it.

Running npm audit

The most immediate step is npm audit. Run this in any Node.js project directory and you will see a report of known vulnerabilities in your dependency tree, organized by severity (critical, high, moderate, low) with links to the CVE details for each.

npm audit fix will automatically update packages where a fix is available and does not introduce a breaking change. npm audit fix --force will apply fixes that may include breaking changes and should be tested carefully before deploying.

Enabling Dependabot on GitHub

Manually running npm audit is useful but requires remembering to do it. Dependabot is a GitHub feature that automatically monitors your repositories for vulnerable dependencies and opens pull requests to update them.

To enable it: go to your repository on GitHub, navigate to Settings, then Security and analysis, and enable Dependabot alerts and Dependabot security updates. That is the entire configuration for basic coverage. You will receive email alerts and automated PRs when vulnerabilities are found.

Pinning Dependency Versions

Using exact version pins in package.json rather than range specifiers reduces the risk of unexpected updates introducing vulnerabilities or breaking changes. The trade-off is that you must manually update dependencies, but you control exactly what is running in production.

Using package-lock.json and committing it to your repository ensures that everyone on your team and your CI system is running exactly the same dependency tree. This is a security measure as much as a consistency measure.

What to Do When You Find a Critical Vulnerability

Check whether the vulnerability affects your actual usage pattern. Many CVEs are listed as critical but only apply when the package is used in a specific way. Read the CVE description. If your usage is affected, update immediately and deploy. If your usage is not affected, document why and add a note to your audit process.

If no fix is available, assess whether the package can be replaced with an alternative. An unfixed vulnerability in a package you are actively using is a real risk that needs to be managed, not deferred indefinitely.

The Baseline Standard

At minimum, every production web application should have Dependabot or an equivalent monitoring active, with alerts routed to someone who will act on them. Running npm audit before every production deployment catches issues that automated monitoring might miss in the window between scans.

Security is not a state you achieve. It is a practice you maintain. Dependency management is one of the most high-leverage parts of that practice because it is largely automated once the systems are in place.
